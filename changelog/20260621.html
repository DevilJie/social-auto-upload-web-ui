01 / CORE · 启动补全

启动时批量补全视频时长

服务启动后在后台 daemon 线程扫描 duration≤0 的视频素材,逐条识别写库;先 sleep 让 Waitress 起来,避免与启动期高频请求争抢 DB。

  • 后台 daemon 线程 在 __main__ DB 初始化后启动 daemon 线程执行,不阻塞 Waitress 端口监听;先 sleep 2 秒让服务先把常规请求处理起来,体验无感知。
  • 精准扫描范围 SQL 过滤 file_type='video' AND (duration IS NULL OR duration<=0),只处理缺失时长的视频;图片素材、已有正常时长的视频全部跳过,零额外开销。
  • 逐条识别写库 每条视频经 resolve_material_path 定位文件 → ffprobe/ffmpeg 识别 → UPDATE duration 写库;识别失败(文件丢失/损坏)不抛异常,跳过继续。
  • 进度日志可见 逐条打印 (1/N) 正在识别: xxx.mp4、识别成功 → 187.6s、最终汇总 共 N 个成功 X 失败 Y;启动过程对补全进展完全透明。

02 / SAFETY · 三层兜底

三层兜底机制

启动补全是异步的,存在时间窗口;因此叠加素材库选中即时 probe 与发布提交同步兜底,任意一层命中都能补全,确保校验拿到真实时长。

  • 第一层:启动批量补全(异步) 服务启动时后台扫描全库,批量识别写库;覆盖面最广,但是异步执行,刚启动时可能尚未轮到某条素材。
  • 第二层:素材库选中即时 probe 沿用 v0.9.0 的 /probe 逻辑,用户在素材库选中某条视频时,若 duration=0 同步识别补全;即时响应,零额外开销(已补全的不会命中)。
  • 第三层:发布提交同步兜底 postVideo 校验入口新增 ensure_duration_or_probe:提交前若发现 duration≤0,立即同步识别补全再校验;前两层都没轮到时的最后防线。
  • 并发去重 三层与上传时的 _async_probe_duration 通过全局 inflight 锁集合去重,避免对同一条素材重复 probe(下载 S3/起子进程开销不小)。

03 / STORAGE · S3 统一

本地存储 + S3 云存储统一处理

S3/S3 兼容协议(含 MinIO、阿里 OSS)只返回文件大小,无法直接拿到视频时长;通过 resolve_material_path 自动下载临时文件后统一交给 ffprobe。

  • 统一路径解析 本地素材直接返回路径;S3 素材经 S3Storage.get_local_path() 优先查本地缓存,无缓存则下载到临时文件,对上层透明。
  • 无 S3 时长接口的解法 S3 协议本身不解析视频容器;复用既有下载逻辑把对象落到本地临时文件,再用 ffprobe 读 duration,无需为云存储单独写识别代码。
  • ffmpeg fallback 时长识别在 ffprobe 之后 fallback 到 ffmpeg -i,处理非标准编码或缺失 metadata 的视频,识别率更高。

存储适配

  • 本地存储直接识别
  • S3 自动下载临时文件
  • 兼容 MinIO / OSS
  • ffprobe + ffmpeg 双引擎
04 / OBSERVABILITY · 日志

进度日志 + 项目统一日志体系

逐条打印识别进度,使用项目统一的 get_channel_logger 写入 data/logs/{日期}/backend.log;不混用 loguru,确保日志在项目既定位置可见。

  • 统一日志体系 使用 util/_logger.py 的 get_channel_logger('backend'),写入 data/logs/{日期}/backend.log,格式与项目其他日志完全一致,不依赖 loguru 控制台输出。
  • 逐条进度 每条视频打印 (N/M) 正在识别、识别成功 → X.Xs 或识别失败;结尾汇总 共 N 个成功 X 失败 Y 跳过 Z,启动过程进度透明。
  • 异常容错 单条识别失败只 warning 不中断;后台任务整体异常绝不把主进程拖崩;提交兜底失败也不影响发布流程(降级为原行为)。

ROOT CAUSE · 问题溯源

为什么会漏识别?

时长识别原本有三条路径,但草稿/历史恢复走数据库直读,完美绕过了识别环节,导致存量 duration=0 的视频成为「漏网之鱼」。

  • ① 上传时识别 ✓ 视频上传后台 _async_probe_duration 异步识别 duration 写库,正常上传的素材有时长。
  • ② 素材库选中时 probe ✓ 用户在素材库选中视频时,前端对 duration=0 的同步调 /probe 补全,选中即有时长。
  • ③ 草稿/历史恢复 ✗ 漏洞所在 草稿框、发布历史恢复走的是数据库直读,直接拿 videoData,根本不经过「素材库选中」,于是 duration=0 的存量数据一路漏到发布提交。
  • ④ 校验被跳过 ✗ 后果 PublishCenter 的视频校验逻辑:if (!duration || duration===0) continue。时长为 0 直接跳过该校验,发布时才发现问题已经晚了。
  • ✓ 本次解法 启动时批量补全存量 + 发布提交同步兜底,无论从哪条路径进来,都能拿到真实时长。
